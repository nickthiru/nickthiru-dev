# Product Content & Build Logs

## Content Creation Context

**For detailed voice specifications, storytelling frameworks, and authentic writing guidance, see the comprehensive style guide in `/src/content/posts/AGENTS.md`.**

This file provides product-specific writing standards, build log frameworks, and product development storytelling guidelines. Use these in combination with the voice and style guidelines from the comprehensive style guide.

## Product Content Focus Areas

### Primary Topics

- **Build Logs**: Daily/weekly development progress and milestones
- **Feature Development**: Specific features built and technical decisions
- **Product Updates**: New releases, improvements, and iterations
- **User Feedback**: Customer insights and how they shaped the product
- **Development Learnings**: Technical lessons and architectural decisions
- **Product Strategy**: Roadmap updates and strategic pivots

### Target Audience

- **Potential users** of Thiru AI Labs products
- **Technical peers** interested in AI product development
- **Other builders** learning from product development process
- **Customers** wanting to understand product evolution
- **Collaborators** and potential partners

## Product Content Standards

### Structure Framework

#### Build Log Posts

1. **Current State**: What I'm working on right now
2. **Today's Progress**: Specific milestone or feature completed
3. **Technical Details**: How it works and key decisions
4. **User Value**: Why this matters to users
5. **Challenges**: What was difficult or surprising
6. **Next Steps**: What's coming next

#### Feature Launch Posts

1. **Problem Context**: User problem or need being addressed
2. **Development Journey**: How the feature was built
3. **Technical Implementation**: How it works under the hood
4. **User Experience**: How users interact with it
5. **Early Feedback**: What beta users are saying
6. **Future Plans**: How this fits into broader roadmap

#### Product Update Posts

1. **What's New**: Specific changes and improvements
2. **Why It Matters**: User benefits and problem solving
3. **Behind the Scenes**: Development insights and decisions
4. **User Impact**: How this affects current users
5. **Metrics & Learning**: Usage data and key insights
6. **What's Next**: Upcoming features and improvements

### Writing Style for Product Content

#### Tone & Approach

- **Development-focused**: Emphasize building and learning process
- **User-centric**: Always connect features to user value
- **Transparent about challenges**: Share difficulties and setbacks
- **Technical but accessible**: Explain complex concepts clearly
- **Progress-oriented**: Show momentum and forward movement

#### Language Patterns

- **Development terminology**: Use appropriate technical language naturally
- **User benefit framing**: Connect technical features to user problems
- **Progress indicators**: Show movement and momentum
- **Learning moments**: Share insights and discoveries
- **Future orientation**: Always point to what's next

## Product Content Guidelines

### Build Log Standards

- **Daily consistency**: Regular updates build trust and momentum
- **Specific progress**: Concrete features and milestones, not vague updates
- **Technical depth**: Explain implementation decisions and trade-offs
- **User connection**: Always connect technical work to user value
- **Honest challenges**: Share difficulties and what you learned

### Feature Launch Standards

- **Problem-first**: Start with user problem, not technical solution
- **Clear benefits**: Explain why users should care
- **Technical transparency**: Share how it works without overwhelming
- **Social proof**: Include early user feedback when available
- **Clear call-to-action**: Tell people exactly what to do next

### Update Communication Standards

- **Value-focused**: Emphasize user benefits over technical details
- **Context provided**: Explain why changes were necessary
- **Impact clear**: How does this affect existing users
- **Future visible**: Show continued development and momentum
- **Feedback invited**: Encourage user input and suggestions

## Quality Checklist

### Before Publishing Product Content

- [ ] **User value**: Is the benefit to users clear and compelling?
- [ ] **Technical accuracy**: Are implementation details correct?
- [ ] **Progress shown**: Does this demonstrate forward momentum?
- [ ] **Learning included**: Are there insights or discoveries shared?
- [ ] **Next steps indicated**: Is there a sense of continued development?
- [ ] **Engagement prompt**: Is there a question or call-to-action?

### Product Impact Assessment

- [ ] **User understanding**: Will users understand what changed and why?
- [ ] **Technical credibility**: Does this demonstrate technical competence?
- [ ] **Trust building**: Does this increase confidence in the product?
- [ ] **Community engagement**: Does this encourage user participation?
- [ ] **Business alignment**: Does this support broader product strategy?
