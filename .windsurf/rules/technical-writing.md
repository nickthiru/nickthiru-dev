## Engineering Content Standards

### Structure Framework

#### Technical Tutorial Posts

1. **Problem Context**: Technical challenge or knowledge gap
2. **Failed Attempts**: What didn't work and why (builds trust)
3. **Breakthrough**: Key insight that changed the approach
4. **Step-by-Step Solution**: Implementation with code examples
5. **Testing & Validation**: How to verify it works
6. **Production Considerations**: Performance, scaling, monitoring
7. **Lessons Learned**: Technical insights and broader principles

#### Architecture Deep-Dive Posts

1. **System Requirements**: Problem constraints and goals
2. **Design Decisions**: Architecture choices and trade-offs
3. **Implementation Details**: How it's actually built
4. **Performance Characteristics**: Benchmarks and limitations
5. **Scalability Considerations**: How it grows with load
6. **Operational Aspects**: Deployment, monitoring, maintenance
7. **Evolution Path**: How the architecture might change

#### Debugging & Troubleshooting Posts

1. **Symptom Description**: What was going wrong
2. **Investigation Process**: How I diagnosed the issue
3. **Root Cause Analysis**: The underlying technical problem
4. **Solution Implementation**: How I fixed it
5. **Verification Steps**: How I confirmed the fix worked
6. **Prevention Measures**: How to avoid this issue
7. **Related Patterns**: Other similar issues to watch for

### Writing Style for Engineering Content

#### Tone & Approach

- **Technical precision**: Use accurate terminology and concepts
- **Educational focus**: Explain complex ideas clearly
- **Practical orientation**: Focus on real-world applications
- **Honest about complexity**: Don't oversimplify difficult concepts
- **Experience-based**: Share from actual implementation work

#### Technical Standards

- **Code accuracy**: All code examples tested and functional
- **Version specificity**: Mention specific versions when relevant
- **Environment context**: Describe the technical environment
- **Error handling**: Include error scenarios and solutions
- **Performance considerations**: Address speed, memory, and scalability

## Technical Content Guidelines

### Code Standards

- **Test all examples**: Every code snippet must be functional
- **Include imports**: Show necessary dependencies and setup
- **Error handling**: Demonstrate proper error handling
- **Comments**: Add explanatory comments for complex code
- **Version info**: Specify relevant versions of libraries/frameworks

### Technical Accuracy

- **Verify claims**: Double-check technical statements
- **Cite sources**: Reference official documentation when appropriate
- **Update regularly**: Keep content current with technology changes
- **Acknowledge limitations**: Be honest about what approaches don't scale
- **Provide context**: Explain when certain solutions are appropriate

### Educational Quality

- **Progressive disclosure**: Introduce concepts step by step
- **Multiple perspectives**: Show different ways to understand concepts
- **Real examples**: Use actual code from production when possible
- **Common pitfalls**: Highlight frequent mistakes and how to avoid them
- **Further reading**: Provide resources for deeper learning

## SEO Integration for Technical Content

### Technical SEO

- **Code snippet optimization**: Use proper syntax highlighting
- **Technical keywords**: Include relevant technology terms naturally
- **Problem-solution format**: Structure content around user problems
- **Long-tail technical queries**: Target specific technical questions
- **Internal linking**: Connect to related technical content

### Content Optimization

- **Meta descriptions**: Include technical keywords and benefits
- **Heading structure**: Use H1, H2, H3 for technical concepts
- **Image alt text**: Describe technical diagrams and screenshots
- **URL structure**: Include technical keywords in URLs
- **Loading speed**: Optimize code examples and technical images

## Quality Checklist

### Before Publishing Technical Content

- [ ] **Code accuracy**: All examples tested and functional
- [ ] **Technical precision**: Correct terminology and concepts
- [ ] **Educational value**: Clear explanations and learning progression
- [ ] **Practical applicability**: Real-world relevance and utility
- [ ] **Error handling**: Covers edge cases and failure scenarios

### Technical Review Standards

- [ ] **Peer review**: Have another technical person review content
- [ ] **Version testing**: Test with different versions of dependencies
- [ ] **Performance validation**: Verify performance claims
- [ ] **Security review**: Check for security implications
- [ ] **Documentation completeness**: All steps clearly explained
