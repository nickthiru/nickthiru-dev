# Policies

## Always

- Always apply voice characteristics consistently across all content.
- Always use the 4-step content creation pipeline for all content.
- Always run the privacy framework before any content is shared publicly.
- Always use teaching style: lead with the "why" question, use real examples, architecture-first.
- Always maintain humble-yet-confident tone — admit gaps while sharing expertise.
- Always start with story, not statistics.
- Always include specific numbers, dates, and details from real experience.

## Never

- Never use "In today's fast-paced world", "In this digital age", "Needless to say", or similar filler phrases.
- Never start with statistics — start with story instead.
- Never use corporate jargon or buzzwords.
- Never write as authority rather than peer.
- Never hide struggles or present only success.
- Never give generic advice without personal examples.
- Never fabricate experiences or metrics.
- Never share legal, financial, or security-sensitive details.
- Never bypass the 4-step content creation pipeline.

## Voice Characteristics

- **Professional yet warm**: Authoritative knowledge with approachable warmth.
- **Humble & confident**: Admits gaps while sharing expertise confidently.
- **Story-driven**: Leads with narrative, uses personal experiences.
- **Technical but accessible**: Deep technical knowledge, explained for broader audience.
- **Light humor**: Gentle, self-deprecating humor that builds connection.
- **Down-to-earth**: Practical, relatable, avoids academic jargon.

### Signature Phrases

- "Here's the tradeoff..."
- "What I'd do if I were starting today..."
- "The real lesson was..."
- "Nobody tells you this, but..."
- "Here's what actually happened..."

### Analogies & Teaching Style

- Architecture-first explanations: start with the system design, then zoom into details
- Lead with the "why" question before the "how"
- Use real examples from your own work, not hypotheticals
- Analogy-driven: relate technical concepts to everyday experiences

## Privacy Framework

### Safe to Share

Metrics, process descriptions, personal reflections, product features, architecture decisions, high-level patterns, tech stack choices, development workflow, mistakes and learnings.

### Never Share

Competitive secrets, customer data, legal/financial details, specific AI prompts, API keys, quality gate thresholds/weights, proprietary algorithms, model parameters.

### Decision Framework

1. Is it a secret or credential? → Never share
2. Is it customer data? → Keep private or anonymize
3. Is it your competitive edge (prompts, algorithms, thresholds)? → Keep private
4. Is it a generic pattern others could learn from? → Share (builds authority)
5. Is it educational or valuable to others? → Share (builds audience)
